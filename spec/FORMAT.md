# spanformat-multilang formatting specification

This document is the normative specification for every language implementation
in this repository. It is derived from the Go reference implementations
[`github.com/apstndb/spantype`](https://github.com/apstndb/spantype) (type
formatting) and [`github.com/apstndb/spanvalue`](https://github.com/apstndb/spanvalue)
(value formatting). When this document and the shared conformance data under
`testdata/` disagree, the conformance data (generated by the Go reference
implementation via `tools/gen-testdata`) wins.

Implementations MUST pass every case in `testdata/conformance.json`.

## 1. Input model

Formatting is defined on the Cloud Spanner wire representation, which every
official client library exposes or can expose:

- **Type**: `google.spanner.v1.Type`
  - `code` (`google.spanner.v1.TypeCode`)
  - `array_element_type` (`Type`, set when `code == ARRAY`)
  - `struct_type.fields` (list of `{name, type}`, set when `code == STRUCT`)
  - `proto_type_fqn` (string, set when `code` is `PROTO` or `ENUM`)
  - `type_annotation` (`google.spanner.v1.TypeAnnotationCode`; PostgreSQL
    dialect markers `PG_NUMERIC`, `PG_JSONB`, `PG_OID`)
- **Value**: `google.protobuf.Value`, encoded by Spanner as follows
  (see the `TypeCode` proto documentation):

| TypeCode | Wire encoding (`google.protobuf.Value` kind) |
|---|---|
| any type, SQL `NULL` | `null_value` |
| `BOOL` | `bool_value` |
| `INT64`, `ENUM` | `string_value`, decimal integer |
| `FLOAT32`, `FLOAT64` | `number_value`; or `string_value` `"NaN"` / `"Infinity"` / `"-Infinity"` |
| `STRING` | `string_value` |
| `BYTES`, `PROTO` | `string_value`, standard (RFC 4648) base64 with padding |
| `TIMESTAMP` | `string_value`, RFC 3339 UTC with `Z` offset |
| `DATE` | `string_value`, `YYYY-MM-DD` |
| `NUMERIC` | `string_value`, canonical decimal (PG_NUMERIC may also be `"NaN"`) |
| `JSON` | `string_value`, normalized JSON text (also used for PG_JSONB) |
| `INTERVAL` | `string_value`, ISO 8601 duration |
| `UUID` | `string_value` |
| `ARRAY` | `list_value` of element encodings |
| `STRUCT` | `list_value` of field value encodings, in field order |

A missing/nil `Value` MUST be treated as SQL `NULL`.

In the shared testdata, `type` is the protojson encoding of `Type` (enums may
appear as **names or numbers**; implementations of the testdata loader must
accept both) and `value` is the protojson encoding of `google.protobuf.Value`,
i.e. a plain JSON value.

## 2. Type formatting (spantype)

`FormatType(type, option)` renders a `Type` as a string. `option` has five
axes; each axis is an enum.

### 2.1 Option axes

- **StructMode**: `BASE` (render `STRUCT`), `RECURSIVE`
  (`STRUCT<INT64, STRUCT<INT64>>`), `RECURSIVE_WITH_NAME`
  (`STRUCT<n INT64, s STRUCT<n INT64>>`).
- **ProtoEnumMode** (separately configurable for PROTO and for ENUM):
  `BASE` (`PROTO` / `ENUM`), `LEAF` (FQN after the last `.`, e.g. `Book`),
  `FULL` (the FQN, e.g. `examples.Book`), `LEAF_WITH_KIND` (`PROTO<Book>`),
  `FULL_WITH_KIND` (`PROTO<examples.Book>`; matches
  `INFORMATION_SCHEMA.COLUMNS.SPANNER_TYPE`).
- **ArrayMode**: `BASE` (`ARRAY`), `RECURSIVE` (`ARRAY<INT64>`).
- **UnknownMode** — rendering of a `code` with no known enum name:
  `UNKNOWN` (`UNKNOWN`), `TYPE_CODE` (decimal of the code, e.g. `999`),
  `VERBOSE` (`UNKNOWN(999)`), `PANIC` (raise an error/panic).
- **TypeAnnotationMode**: `SUFFIX` (default; append `(<annotation name>)` when
  the annotation is not `TYPE_ANNOTATION_CODE_UNSPECIFIED`, e.g.
  `NUMERIC(PG_NUMERIC)`), `OMIT` (ignore annotation), `PRIMARY` (when the
  annotation is set, render **only** the annotation name, e.g. `PG_NUMERIC`,
  with no recursion into the base type).

### 2.2 Algorithm

```
FormatType(t, opt):
  ann = t.type_annotation
  if opt.type_annotation == OMIT:      return impl(t, opt)
  if opt.type_annotation == PRIMARY:   return ann_name       if ann set
                                       return impl(t, opt)   otherwise
  # SUFFIX (default)
  return impl(t, opt) + ("(" + ann_name + ")" if ann set else "")

impl(t, opt):
  switch t.code:
    ARRAY  where opt.array == RECURSIVE:
        return "ARRAY<" + FormatType(t.array_element_type, opt) + ">"
    PROTO: return proto_enum(t, opt.proto)
    ENUM:  return proto_enum(t, opt.enum)
    STRUCT where opt.struct != BASE:
        return "STRUCT<" + join(fields, ", ") + ">"
        # each field: FormatType(f.type, opt); prefixed by "<name> " when
        # opt.struct == RECURSIVE_WITH_NAME and f.name != ""
  return type_code_name(t.code, opt.unknown)
```

`proto_enum` renders per ProtoEnumMode above; `BASE` renders the enum name of
the code (`PROTO`/`ENUM`). The type-annotation rule applies at **every**
recursion level (each nested `FormatType` call re-applies it).

`type_code_name` renders the `TypeCode` enum name
(`TYPE_CODE_UNSPECIFIED`, `BOOL`, `INT64`, `FLOAT64`, `FLOAT32`, `TIMESTAMP`,
`DATE`, `STRING`, `BYTES`, `ARRAY`, `STRUCT`, `NUMERIC`, `JSON`, `PROTO`,
`ENUM`, `INTERVAL`, `UUID`), falling back to UnknownMode for unknown numbers.
Note `ARRAY`/`STRUCT`/`PROTO`/`ENUM` reach this fallback when their mode is
`BASE` — the enum name is rendered.

### 2.3 Presets

| Preset | Struct | Proto/Enum | Array | Unknown |
|---|---|---|---|---|
| `SIMPLEST` | BASE | BASE | BASE | TYPE_CODE |
| `SIMPLE` | BASE | LEAF | RECURSIVE | UNKNOWN |
| `NORMAL` | RECURSIVE | LEAF | RECURSIVE | VERBOSE |
| `VERBOSE` | RECURSIVE_WITH_NAME | FULL | RECURSIVE | VERBOSE |
| `MORE_VERBOSE` | RECURSIVE_WITH_NAME | FULL_WITH_KIND | RECURSIVE | VERBOSE |

All presets default TypeAnnotationMode to `SUFFIX`.

## 3. Value formatting (spanvalue)

`FormatValue(type, value, config)` renders one column value. `config` selects
one of three presets plus options:

| Preset | Null string | Purpose |
|---|---|---|
| `SIMPLE` | `<null>` | Human-readable, Go client library conventions |
| `LITERAL` | `NULL` | Re-parseable GoogleSQL literal expressions |
| `SPANNER_CLI` | `NULL` | Byte-for-byte [spanner-cli](https://github.com/cloudspannerecosystem/spanner-cli) output |

Options: `null_string` (override the preset's null string; MUST NOT be empty)
and, for `LITERAL` only, a quote policy (§3.6).

Formatting is recursive over `(Type, Value)` pairs with a boolean `toplevel`
flag (`true` for the column itself, `false` inside ARRAY elements and STRUCT
fields).

### 3.1 NULL

If the value is SQL `NULL` (`null_value` or missing), render the null string —
at any nesting depth, for any type, in every preset.

### 3.2 ARRAY (non-NULL)

Format each element recursively with `toplevel=false`, join with `", "`:

- `SIMPLE`, `SPANNER_CLI`: `[e1, e2, ...]`
- `LITERAL`: `[e1, e2, ...]`, prefixed with the `VERBOSE` type-format of the
  whole array type **iff** `toplevel` and the element type code is `ARRAY` or
  `STRUCT` (regardless of element count):
  `ARRAY<STRUCT<n INT64>>[(1)]`, `ARRAY<STRUCT<n INT64>>[]`. Scalar-element
  arrays never get the prefix: `[1, 2]`, `[]`.

### 3.3 STRUCT (non-NULL)

The number of wire values MUST equal the number of type fields (else error).
Format each field value recursively with `toplevel=false` using the field's
type, then:

- `SIMPLE`: each field rendered as `<value> AS <name>` when the field name is
  non-empty, else just `<value>`; joined with `", "`; wrapped in `(...)`.
- `LITERAL`: field values only, joined with `", "`, wrapped in `(...)`;
  when `toplevel`, prefixed with the `VERBOSE` type-format of the struct type:
  `STRUCT<n INT64, s STRING>(1, "foo")`. Empty struct at toplevel:
  `STRUCT<>()`.
- `SPANNER_CLI`: field values only, joined with `", "`, wrapped in `[...]`
  (bracket style): `[1, foo]`.

### 3.4 Scalars

Wire-payload validation: a scalar whose value kind does not match §1's table
is an error (malformed wire), except as noted for floats.

| TypeCode | SIMPLE | LITERAL | SPANNER_CLI |
|---|---|---|---|
| `BOOL` | `true` / `false` | same | same |
| `INT64` | wire string as-is | wire string (MUST parse as int64, else error) | wire string as-is |
| `FLOAT32`/`FLOAT64` | §3.5 go-g shortest | §3.5 literal | §3.5 CLI |
| `STRING` | wire as-is | quoted + escaped (§3.6, §3.7) | wire as-is |
| `BYTES` | readable bytes (§3.8) of decoded base64 | `b"..."` bytes literal (§3.6, §3.7) | base64 wire as-is |
| `TIMESTAMP` | wire as-is | `TIMESTAMP "<wire>"` (string literal per §3.6/§3.7) | wire as-is |
| `DATE` | wire as-is | `DATE "<wire>"` | wire as-is |
| `NUMERIC` | wire as-is | `NUMERIC "<wire>"` | wire with trailing fractional zeros trimmed (§3.9) |
| `JSON` | wire as-is | `JSON "<wire>"` (escaped) | wire as-is |
| `INTERVAL` | wire as-is | `CAST("<wire>" AS INTERVAL)` | wire as-is |
| `UUID` | wire as-is | `CAST("<wire>" AS UUID)` | wire as-is |
| `ENUM` | wire as-is (decimal string) | `CAST(<wire> AS \`<fqn>\`)` (wire MUST parse as int64; `proto_type_fqn` MUST be non-empty) | wire as-is |
| `PROTO` | readable bytes of decoded base64 | `CAST(b"..." AS \`<fqn>\`)` (bytes literal per §3.6/§3.7; fqn MUST be non-empty) | base64 wire as-is |

`TYPE_CODE_UNSPECIFIED` and unknown codes are errors in all presets.
The `NUMERIC`/`JSON` rules apply identically to their `PG_NUMERIC`/`PG_JSONB`
annotated forms (e.g. PG_NUMERIC `NaN` renders as wire string `NaN` in SIMPLE
and `NUMERIC "NaN"` in LITERAL).

In `TIMESTAMP "<wire>"` etc. above, the double quotes stand for a §3.6/§3.7
string literal of the wire payload — the delimiter follows the quote policy.

### 3.5 Float formatting

Wire reading: `number_value` → the double; narrowed to float32 first for
`FLOAT32`. `string_value` must be exactly `NaN` / `Infinity` / `-Infinity`;
anything else is an error.

**go-g shortest** (used by SIMPLE, and by LITERAL for finite values) renders
exactly like Go's `strconv.FormatFloat(v, 'g', -1, bits)`:

1. Compute the shortest decimal digit string `d1 d2 … dn` (no trailing zeros)
   and decimal exponent `exp` (value = `d1.d2…dn × 10^exp`) such that parsing
   back at the same bit width (32 or 64) recovers the value exactly.
   Portable technique: for `p` in `0..16` (float64) / `0..8` (float32), format
   with C-style `%.{p}e` and stop at the first `p` whose result round-trips.
2. If `exp < -4` or `exp >= 6`: scientific form
   `d1[.d2…dn]e±XX` — no decimal point when `n == 1`; exponent has a sign and
   at least two digits (`1e+06`, `5e-324`, `1.2345678901234568e+20`).
3. Otherwise: positional form without exponent and without extra zeros
   (`100000`, `0.0001`, `123.456`).
4. Negative values get a `-` prefix; negative zero renders `-0`.

**LITERAL**:
- Finite `FLOAT64`: go-g shortest; if the result contains none of `.`, `e`,
  `E`, append `.0` (`1.0`, `100000.0`, but `1e+06` stays).
- Finite `FLOAT32`: `CAST(<go-g shortest at 32-bit> AS FLOAT32)`.
- Non-finite: `CAST("nan" AS FLOAT64)`, `CAST("inf" AS FLOAT64)`,
  `CAST("-inf" AS FLOAT64)` (or `FLOAT32`), where the quoted payload is a
  §3.6 string literal of `nan` / `inf` / `-inf` (delimiter follows the quote
  policy; these payloads contain no quotes so legacy/min-escape use the
  preferred delimiter).

**SPANNER_CLI**: finite values with zero fractional part (`v == trunc(v)`)
render as C-style `%.0f` (`3`, `-42`); other finite values as `%.6f`
(`3.141593`, `0.500000`); non-finite render `NaN`, `+Inf`, `-Inf`.
`%.Nf` means correctly-rounded fixed-point of the double value (IEEE
round-half-even at the binary level, like C `printf`).

### 3.6 Quote policy (LITERAL string/bytes literals)

A quote policy is `(strategy, preferred)`:

- `strategy`: `LEGACY` (default), `ALWAYS`, `MIN_ESCAPE`
- `preferred`: `DOUBLE` (default, `"`), `SINGLE` (`'`)

Delimiter selection for a payload (string: its UTF-8 bytes; bytes: raw bytes;
only the ASCII bytes `'` and `"` matter):

- `ALWAYS`: always the preferred delimiter.
- `LEGACY`: if the payload contains the **non-preferred** quote character →
  preferred; else if it contains the preferred quote character → the other
  delimiter; else preferred. (I.e. switch away from preferred only when that
  avoids all quote escapes.)
- `MIN_ESCAPE`: count `'` and `"` occurrences; the delimiter with fewer
  occurrences in the payload wins; tie → preferred.

The chosen delimiter is then used with §3.7 escaping. This policy applies to:
STRING literals, BYTES/PROTO `b"..."` literals, the string payloads of
`TIMESTAMP/DATE/NUMERIC/JSON` literals and `INTERVAL/UUID` casts, and the
`nan`/`inf`/`-inf` payloads of non-finite float casts.

### 3.7 Escaping (GoogleSQL)

Given the chosen delimiter `q`, each **rune** of a STRING (or each **byte**,
treated as a code point 0–255, of a BYTES/PROTO payload) is escaped:

1. `q` itself or `\` → `\` + the character.
2. (STRING only) `\n` → `\n`, `\r` → `\r`, `\t` → `\t`.
3. (STRING only) if the rune is *printable* — Unicode categories L, M, N, P, S,
   or U+0020 SPACE (Go `unicode.IsPrint`) — emit it as-is.
4. Else if `0x20 <= r <= 0x7E` — emit as-is (reachable only on the BYTES path).
5. Else if `r < 0x100` → `\xHH` (two lowercase hex digits).
6. Else if `r > 0xFFFF` → `\UHHHHHHHH` (eight lowercase hex digits).
7. Else → `\uHHHH` (four lowercase hex digits).

String literals are `q payload q`; bytes literals are `b` + `q payload q`.

### 3.8 Readable bytes (SIMPLE BYTES/PROTO)

Decode the base64 wire. If every byte is in `0x20..0x7E` and is not `\`,
return the raw bytes as a string (empty input → empty string). Otherwise
escape every byte with §3.7 rules using *no* delimiter (rule 1 applies only to
`\`; rules 2–3 do not apply; so: `\` → `\\`, `0x20..0x7E` as-is, else `\xHH`).

### 3.9 Spanner CLI NUMERIC trimming

If the wire string contains a `.`: strip trailing `0`s, then a trailing `.`.
(`3.140` → `3.14`, `10.0` → `10`, `10` → `10`.)

## 4. API surface (per language, adapted to idiom)

Required, under names idiomatic to the language:

- Type formatting: `format_type(type, option)`, preset helpers
  (`format_type_simplest|simple|normal|verbose|more_verbose`).
- Value formatting: a `FormatConfig` (or equivalent) constructed from presets
  `simple` / `literal` (with optional quote policy) / `spanner_cli`, with a
  `null_string` override; `format_value(type, value)` for one column
  (toplevel) and `format_row` over a list of columns.
- Errors are reported per the language's convention (exceptions / Result /
  error codes); the error classes to distinguish are *malformed wire*,
  *unknown type*, and *mismatched struct fields*.

Public API SHOULD accept the official client library's `Type`/`Value`
representations (structurally / duck-typed where that avoids a hard
dependency), and MUST be documented with a working example of formatting a
query result row obtained from the official client library.
